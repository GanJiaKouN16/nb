# AI编程Token节省技巧

--notebook Other

AI编程Token节省技巧

核心心法：把AI当作按字数收费的顶级外包专家——需求给准、资料给对、废话不说。

一、Token计费的两个核心真相
1. 上下文的"滚雪球"效应（Input Token）
大模型是无状态的，每次请求必须把对话历史、引用的代码文件、系统提示词全部打包发送。聊得越久，每一句话的成本越高。

2. 输出比输入贵3-4倍（Output Token）
在大多数高级模型计费中，Output Token价格通常是Input Token的3-4倍。让AI说废话是最昂贵的浪费。

省钱三维度：输出端（禁止废话）+ 输入端（控制上下文长度）+ 处理端（按任务难度分级）。

二、6大实战技巧
技巧1：Rules（建立输出防御边界）
在Rules中写入硬性要求，每次对话自动带入：

输出约束：纯净交付（禁README/文档/总结）；默认无测试；禁止复述需求

交互风格：零客套（禁"好的""我明白"）；直接给唯一最优解；不提供A/B方案

分场景解释策略：

简单修改（改变量名、修小Bug）→ 零解释，直接给代码

新增功能 → 代码块上方用1-2句简述设计思路

关键决策（安全/性能/架构）→ 必须简要说明理由

执行原则：最小修改（禁擅自重构）；歧义时仅提1个关键问题，禁猜测

技巧2：需求拆分（一次性说清楚）
痛点：挤牙膏式对话让AI反复生成代码，Token指数级增长

对策：一次性把要求、边界条件、参考范例说清楚。少说"不要做什么"，多说"要怎么做"，给出正确示范

Tips：专有技术名词用英文（如 debounce function），复杂逻辑解释用中文

技巧3：技术方案确认（先对齐再执行）
痛点：AI理解偏了会生成大量错误代码，甚至覆盖未提交代码

对策：让AI先用自然语言描述修改计划、涉及文件、核心逻辑，确认无误后再生成代码

磨刀不误砍柴工，确认方案再执行是省钱又省心的关键

技巧4：上下文长度控制（4种方法）
4.1 手动控制：改哪里选哪里，手动选中问题代码块，用 @Files 指定文件

4.2 多对话框：专人专事，用完即走。聊方案用便宜模型，确定方案后新建对话框交给高级模型执行

4.3 压缩上下文：在设置里新建指令 /compact，内容设为"总结当前进度，压缩上下文"

4.4 清空上下文：AI逻辑混乱时，直接清空或新建对话，别试图纠正

技巧5：Agent Skills（按需加载上下文）
原理：把大模型比作全科医生，Skills就是专科医疗包——平时不加载，用时才调用

适合场景：代码冲突解决、生成单元测试、老旧代码重构

配置结构：

name：技能名称

description：功能描述

triggers：触发关键词（如 "git conflict"）

globs：适用文件类型

alwaysApply: false（关键：不全局加载）

Instructions：激活时才加载的详细指令

对比：不用Skills时全局Prompt需一直带着几百字规则，每次对话都要付费；用Skills则平时上下文干净，触发时才加载专家模式

技巧6：模型选择 + Prompt Caching
模型分级：

GPT-5.2等高级模型：仅用于复杂逻辑分析、架构设计、疑难Bug修复

普通/免费模型：用于工具函数、TypeScript类型、样板代码、注释

Prompt Caching：System Prompt、Rules和基础代码库上下文固定时，API厂商会缓存这部分内容，缓存命中的Token价格仅为写入价格的1/10。确保Rules和Reference Files放在对话最前面以触发缓存。

三、优化效果预期
维度    优化前    优化后    预期收益
上下文    全局扫描，动辄50k+ Token    手动引用+压缩，控制在10k内    Input成本降低60%-80%
输出    大量解释、文档、未修改代码    仅输出Diff代码和关键注释    Output成本降低50%+
准确率    上下文污染导致幻觉，反复修改    专人专事（Skills），方案先行    减少返工隐性浪费
