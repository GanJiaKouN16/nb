# Harness 工程化

> **核心观点**：AI 编码的瓶颈已从"模型能力"转向"流程工程"。模型提供智商，而 `harness` 提供纪律。

## 一、核心思想：三层设计堵住"遗忘"根因

Agent 的"遗忘"存在三重根因：
1. **压缩丢失**（上下文窗口有限）
2. **检索失败**（信息未被正确召回）
3. **指令遵循失败**（规范冲突或忽略）

针对以上痛点，`harness` 的核心策略是：
- **规则外置**：不依赖模型记忆，而是写在文件系统中。
- **状态持久化**：中间产物落盘，可追溯、可验证。
- **门禁阻断**：用确定性程序拦截不确定性操作，失败即止。

## 二、分层架构：把上下文当预算管理

核心原则：**"主会话上下文是最贵的资源"**，分层标准并非按功能，而是按"何时被读取"。

| 层级 | 目录/载体 | 核心内容 | 预算控制 |
| :--- | :--- | :--- | :--- |
| **常驻入口层** | `CLAUDE.md` | 角色定义、流程触发规则 | **常驻上下文 ≤ 8K** |
| **原子规则层** | `rules/` | 7个单一职责规则（如禁用 `mvn -am`） | 强制约束，固化踩坑经验 |
| **角色 Agent 层** | `agents/` | 拆分流水线（`dispatcher` 路由 + `orchestrator` 合成） | 主会话退化为纯执行器 |
| **按需上下文层** | `context/` | 完整流程模板 | 像弹药一样，"打仗时才搬上来" |
| **执行支撑层** | `skills/` + `commands/` | 封装研发工具链 | 包含经验三级进化机制 |

### 经验三级进化机制
框架会随着使用越来越聪明，进化路径为：
`lesson（教训） → pattern（模式） → instinct（本能）`

## 三、关键机制：门禁外置与动态裁剪

### 1. 稳定性支点：G1–G8 门禁
- 门禁是 **确定性 Python 函数**，不依赖 LLM 推理。
- 检查点包括：**编译检查**、**单元测试**等。
- 任一失败则 **流程阻断退回**，不继续生成错误代码。
- `hook` 机制实时拦截危险操作（如执行破坏性命令）。

> **核心原则**：流程控制必须从 LLM 推理中外置到确定性基础设施。

### 2. 动态流程裁剪
完整链路包含 **19 个节点**，但实际执行深度由 **"意图 × 风险"** 动态决定：
- `QUERY`（低风险）：几乎不走流程。
- `FEATURE/HIGH`（高风险）：拉满 19 个节点进行全量检查。

## 四、演进教训：四阶段踩坑史

| 阶段 | 策略 | 结果 |
| :--- | :--- | :--- |
| **1. 拿来主义** | 直接使用开源配置 | 无法适配特定工程环境 |
| **2. 重 Prompt 约束** | 写大量规则塞进 Prompt | 约 3 天后系统崩溃（上下文溢出、指令冲突） |
| **3. 减负分层加载** | 按需加载，降低上下文负担 | 效果提升，但调度混乱 |
| **4. Agent 调度编排** | 引入 `dispatcher` 和 `orchestrator` | **最终态**：流程清晰，职责单一 |

> **核心理念转变**：从"用更多的字约束 AI"转变为 **"用更好的结构约束 AI"**。

## 五、评测驱动：把流程当被测对象

为了解决"修改规范后效果变好还是变坏"的争论，构建了评测平台：

### 评分机制
- **纯确定性评分**：**100% Python 逻辑**，零 LLM 调用。
- **七维打分**：涵盖流程完整性、代码正确性等维度。
- **一致性保障**：保证 3 次跑分结果完全一致。

### 防注水机制
- **真编译、真跑单测**：代码正确性不以 AI 自评为准，以实际执行为准。
- **诚实度检测**：计算 AI 自报结果与真实结果的"诚实度差距"。

## 六、总结与迁移边界

### 可迁移的模式
这套"工程化"逻辑可迁移到任何 **"能力够强但输出不稳定、且过程可观测"** 的 AI 工作流中。

### 适用边界
- **依赖前提**：过程必须可观测（中间产物能落盘检测）。
- **时效性**：该套价值可能会随模型基础能力的自我进化而逐渐衰减（当模型本身足够稳定时，工程约束成本变高）。
