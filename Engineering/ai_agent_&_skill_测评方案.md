# AI Agent & Skill 测评方案

---

## 一、背景与痛点

### Agent 带来的三大新问题
1. **非确定性**：同一 prompt 多次执行结果不同
2. **黑盒化**：模型升级、Prompt 微调、工具链变化导致行为漂移，肉眼难察觉
3. **错误级联放大**：一次任务涉及几十步工具调用，前序小偏差沿链路逐级放大

### 无测评的被动局面

| 痛点 | 后果 |
|------|------|
| 主观性强 | 依赖直觉判断，缺乏量化依据 |
| 悄悄退化 | 改 Prompt 或升级依赖，旧场景变差无人知晓 |
| 人工验证成本高 | Skill 越多、迭代越快，人力回归成本指数增长 |
| 模型不敢升级 | 无对比数据支撑切换决策 |
| 缺少效率基线 | 无延迟/Token/费用历史基线，变贵变慢无法归因 |
| 过程易忽略 | 最终答案碰巧正确但推理路径错误，无法区分 |

### 核心理念
> Eval = Agent 输入 → 执行 → 捕获 Trace + 产物 → 一组检查规则 → 可对比的分数

目标：**可重复、可量化、可持续演进**的评估闭环，用数据代替直觉，用全量代替抽查。

## 二、测评框架

### 2.1 三类评委（谁来打分）

| 维度 | 确定性评分器 | Rubric 评分器（LLM-as-Judge） | 人工评分器 |
|------|-------------|------------------------------|-----------|
| **谁来评** | 脚本（Bash/Python） | 大模型（固定版本） | 领域专家 |
| **规则写在哪** | 代码里 | Prompt + JSON Schema | 人脑 + 标注指南 |
| **成本** | 毫秒级 / 免费 | 秒级 / API 费用 | 分钟-小时级 / 人力 |
| **稳定性** | 100% | 有抖动（需降噪） | 取决于标注员水平 |
| **覆盖能力** | 已知硬指标 | 未知软指标 | 主观 + 边缘场景 |
| **典型用例** | 文件存在、构建通过 | 代码风格、意图贴合、解释清晰度 | 校准 LLM、诊断异常、红队测试 |
| **门禁角色** | 硬门禁 | 分级门禁（error 硬/warning 软） | 不进门禁，采样审查 |

**优先级：确定性评分器 > Rubric 评分器 > 人工评分器**

#### 确定性评分器适用场景
- 工具调用检查（是否调用指定工具、参数是否正确）
- 产物检查（文件是否存在、内容是否符合预期）
- 关键词匹配
- 执行指标（工具调用次数、Token 消耗是否在阈值内）
- 基线对比

#### Rubric 评分器适用场景
- LLM 评判回答质量、语气、规范遵循度
- 检查推理过程是否清晰连贯、是否存在幻觉

#### 人工评分器介入的六个必要场景
1. 校准 LLM 评委（抽样 100-200 条，一致率 ≥ 85% 才可用）
2. 主观任务打分（回复同理心、报告论证严谨度）
3. 诊断通过率异常（0% 或 100% 通常是任务/评分器坏了）
4. 建立 Ground Truth（新套件上线前 20-50 个参考解）
5. Trace 采样审查（每周固定抽样）
6. 高风险兜底（医疗/金融/安全场景 100% 人工复核）

> 专家时间应 **60% 以上** 花在"校准 LLM 评委"和"诊断异常"上。

---

### 2.2 五个维度（评什么）

| 大类 | 子维度 | 主要评委 | 优先级 |
|------|--------|---------|--------|
| **1. 功能正确性** | 结果正确性、任务完成度、指令遵循、工具调用正确性 | 代码 | **P0** |
| **2. 过程质量** | 推理合理性、步骤最优性、信息完整性、上下文利用率 | Rubric + 人工 | P1 |
| **3. 效率与成本** | Token消耗、工具调用次数、延迟、失败重试率 | 代码 | P1 |
| **4. 鲁棒性与安全** | 一致性(pass^k)、异常恢复、抗对抗、幻觉率、越权风险、合规性 | 代码 + 人工 | **P0** |
| **5. 体验与对齐** | 语气风格、清晰度、主动澄清、同理心、品牌一致性 | Rubric + 人工 | P2 |

**术语说明**：
- **Rubric**：用结构化的评分提示词让另一个大模型充当评委
- **pass^k**：k 次试验中**每次都**通过的概率（衡量稳定性）
- **pass@k**：k 次中**至少 1 次**通过的概率（衡量峰值能力）

#### 各维度详解

**大类1：功能正确性（P0，必须自动化全覆盖）**
- 结果正确性：代码比对、单元测试 → pass@1 / pass^k
- 任务完成度：子目标打点 → 完成率%
- 指令遵循度：JSON Schema 校验、正则 → 遵循率%
- 工具调用正确性：调用日志断言 → 调用准确率%

**大类2：过程质量（Rubric 主战场）**
- 推理合理性、步骤最优性、信息完整性、上下文利用率、自我纠错能力

**大类3：效率与成本（常被忽视但极其重要）**
- Token 消耗（avg/p95）、工具调用次数（avg/p95）、端到端延迟（p50/p95/p99）、失败重试率、单次任务成本（¥/task）

**大类4：鲁棒性与安全（P0，金融/医疗/企业数据场景必须前置）**
- 一致性/稳定性（pass^k）、异常恢复、抗对抗/抗注入、幻觉率、越权/越界风险、合规性、拒绝合理性

**大类5：体验与对齐（P2，决定产品生死）**
- 语气风格、回复清晰度、主动澄清率、同理心、可解释性、CSAT/NPS

---

### 2.3 不同类型 Agent 的测评侧重

| Agent 类型 | 测评侧重点 | 典型检查项 |
|-----------|-----------|-----------|
| 知识库问答 | 准确性、幻觉检测、引用溯源 | 回答基于知识库、正确引用来源 |
| 代码编写 | 产物正确性、可运行性 | 编译/运行通过、满足功能需求 |
| 功能工具 | 过程合规性、工具调用正确性 | 按预期步骤调用、参数正确 |
| 问题定位 | 推理链路、根因准确性 | 逻辑清晰、定位真实根因 |

## 三、测评实施

### 3.1 设计测评用例集

**四大场景（正向 + 负向）**

| 场景 | 关注点 | 正向用例 | 负向用例 |
|------|--------|---------|---------|
| ① 触发 | Skill 是否在正确场景被激活 | 各种合法表述都能触发 | 相似但不相关的 prompt 不误触发 |
| ② 核心逻辑 | 执行过程是否按预期步骤进行 | 工具调用序列、参数、顺序正确 | 缺少关键步骤、调用错误工具 |
| ③ 产物质量 | 最终产物是否完整、准确、可用 | 响应准确、文件完整、格式正确 | 内容缺失、格式错误、幻觉 |
| ④ 异常容错 | 异常输入/环境下能否合理处理 | 优雅降级、明确提示 | 崩溃、死循环、静默失败 |

**场景二设计方法：三步法**
1. 梳理分支流程图（所有核心分支路径）
2. 每条分支至少 1 个用例
3. 补充组合和边界（分支间组合、阈值边缘、跨分支冲突）

> 核心逻辑用例数量通常是其他三类之和的 **2-3 倍**。分支过多时，优先覆盖**高频分支**和**易出错分支**。

---

### 3.2 设计评分规则

- **负分制**，初始 100 分，扣至 0 分
- 达标标准：**80 分**（推荐值，可调整）

| 评分维度 | 计分项 | 扣分规则 |
|---------|--------|---------|
| 结果 | 任务结果 | 不符合预期扣 100 分 |
| 过程 | 步骤遵循性 | 每步不符合扣 10 分 |
| 过程 | 步骤输出结果 | 每步不符合扣 10 分 |
| 过程 | 调用工具/知识库 | 每项不符合扣 10 分 |
| 效率 | 分析耗时 | 基准 5 分钟，每超 1 分钟扣 10 分 |
| 稳定性 | 一致性 | N 次试验（建议 N=5），全部达标取平均 |

---

### 3.3 建立用例基线

**核心思路**：先跑出来 → 人工确认 → 确认后即为预期基线

**基线内容**：
- **预期过程**：思维链、工具调用序列（含入参和返回值）、中间产物、完整 Trace
- **预期结果**：最终响应、输出文件、输出报告

**基线在评分中的作用**：
- **确定性评分器**：程序化对比（工具调用序列是否一致、产物是否存在、Token 是否超标）
- **Rubric 评分器**：作为参考答案（Reference Answer），模型对比语义差异

**基线更新时机**：
- Agent/Skill 逻辑变更 → 重新执行 1 次确认新基线
- 模型版本升级 → 重新执行 1 次确认新基线
- 用例本身修改 → 重新执行 1 次确认新基线

---

### 3.4 执行测评

**触发时机**：
- Agent/Skill 提示词变更 → 自动触发（每次 PR 合入前）
- 模型版本升级 → 手动触发
- 定期巡检 → 定时触发

---

### 3.5 用例集维护

**能力测评 vs 回归测评**

| 类型 | 核心问题 | 期望通过率 | 作用 |
|------|---------|-----------|------|
| 能力测评 | Agent 能把什么做好？ | 起步 20-50%，逐步提升 | 指导优化方向 |
| 回归测评 | 原有能力是否还在？ | 接近 100% | 防御漂移 |

- 能力测评通过率稳定 = 100% 后，"毕业"到回归套件
- 回归用例集**只加不减**
- 确定性任务通过率需 **100%**，非确定性任务推荐 **≥ 95%**

**Bad Case 处理流程**：
1. 复现 Bad Case，记录完整轨迹
2. 分析根因，确定预期行为
3. 修复 Agent/Skill
4. 将 Bad Case 纳入用例集
5. 通过率稳定后毕业到回归测评集

## 四、工程落地

### 4.1 Trace 输出能力

**对被测 Agent/Skill 的要求**：
- 输出结构化 Trace（JSONL/JSON 格式）
- 至少包含：每步工具名称、入参、返回值、时间戳
- 理想情况还包含思维链和决策依据
- Trace 格式稳定，字段命名跨版本一致

> 建议在 Agent/Skill **设计阶段**就将结构化 Trace 输出作为标准能力纳入。

---

### 4.2 环境隔离
- Clone 仓库到临时目录
- 每个用例执行前重置环境（`git checkout . && git clean -fd`）
- 测评产物统一归档

---

### 4.3 稳定性评估

**核心原则**：N 次中有 1 次不通过，即存在稳定性风险

**容忍阈值**（按 Agent 类型）：

| Agent 类型 | 容忍阈值 | 说明 |
|-----------|---------|------|
| 关键决策类（故障诊断） | **0%** | 任何失败都不可容忍 |
| 辅助分析类（性能分析） | ≤ 10% | 偶发偏差可接受 |
| 创意生成类（代码建议） | ≤ 40% | 多样性本身是特性 |

> **调试技巧**：0% pass^N 通常说明**任务定义有问题**，而非 Agent 能力不行。

---

### 4.4 测评报告

报告应包含：
- **全局概览**：生成时间、用例总数、通过率、平均分、模型版本、总 Token、总费用
- **用例列表**：按任务分组，显示用例数、均分、Trial 分数分布图
- **单用例详情**：基线信息（耗时/Token/费用/步骤数）、Prompt
- **稳定性评分**：N 次执行达标情况 + 平均分
- **执行记录**：每次 Trial 的 ID、Token、费用、耗时、各维度扣分明细

## 五、实战案例：TPerf 性能 AI 分析 Agent

### 5.1 Agent 概述
- 类型：功能工具，部署在 Knot 平台
- 通过 MCP 工具调用 TPerf-API 获取性能数据
- 结合知识库检索，经决策树判断瓶颈，输出结构化报告
- 测评侧重：**操作步骤合规性 + 执行效率 + 输出报告质量**

---

### 5.2 评分标准

**确定性评分（脚本自动判定）**

| 维度 | 最大扣分 | 评分方法 |
|------|---------|---------|
| 操作步骤 | -10 | LCS 算法对齐基线与 Trial 步骤，不一致/多余/缺少/顺序错误各扣 2 分 |
| 效率 | -10 | 耗时以 300s 为基准，每超标 10% 扣 1 分；Token 每超标 10% 扣 1 分 |

**模型评分（LLM 按 Rubric 判定）**

| 维度 | 最大扣分 | 评分方法 |
|------|---------|---------|
| 关键判定 | **-80** | 压测有效性/性能结果与基线不一致直接扣 80 分 |
| 章节内容 | 每项 -5，上限 -80 | CPU/内存/网络/磁盘IO/业务指标/根因/优化建议等逐项对比 |

**一致性判断标准**：
- 数值型：整数精确匹配，小数允许相对误差 ≤ 1%
- 结论型：异常等级和关键发现一致
- 建议型：核心排查方向一致即可

**综合评分**：单次 Trial 达标线 80 分，任一 Trial 不达标则该用例得 0 分

---

### 5.3 用例集设计

按性能瓶颈场景分组，覆盖 9 大类 30+ 用例：
- CPU 瓶颈（6）、网卡瓶颈（5）、磁盘瓶颈（4）、磁盘 IO（2）、内存（1）、TCP 队列（3）、Nginx（1）、配置问题（1）、正常场景（5）

每个用例包含：用例名称和描述、压测记录 ID、触发提示词、基线会话标识

---

### 5.4 基线管理

基线以 Knot 会话标识（session_id + message_id）保存，评测时通过 API 动态获取完整数据。

**基线快照内容**：完整 Markdown 报告、工具调用流程、执行耗时、Token 消耗

---

### 5.5 执行与工程实现

**执行模式**：
- 快速分析：每用例 1 轮，日常调试
- 全量测评：每用例 5 轮 × 7 模型，版本验收
- 按模型筛选、重新评分

**Trace 获取**：通过 Knot API 轮询获取 AG-UI 事件流（工具调用事件、文本消息、时间消耗）

**框架分层**：API 封装层、配置层、用例定义层、采集层、评分层、执行层、解析层、结果层

**流水线**：
- 基线确认流水线：日常开发调试
- 测评流水线：pytest 实现，并发执行（Trial 并发 + 评分并行 + 三评分器并行）

## 附录：术语速查

| 术语 | 含义 |
|------|------|
| **Prompt** | 提示词，发送给 AI 模型的指令文本 |
| **Token** | 词元，模型处理文本的最小单位（约 0.7 个中文字或 0.75 个英文单词） |
| **Trace** | 执行轨迹，Agent 执行过程中产生的结构化日志，记录每一步的工具调用、参数、返回值和思考过程 |
