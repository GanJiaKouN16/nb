## 〇、创作前：Agent 输入规范

在开始创作前，Agent 应向用户确认以下信息：

### 必填项

| 项目 | 说明 | 示例 |
|:-----|:-----|:-----|
| **听众** | 谁会看/听这个演示？ | 高管团队、技术同事、客户 |
| **时长** | 汇报时间（不含讨论） | 10 分钟、30 分钟 |
| **核心诉求** | 演示的根本目的是什么？ | 说服批准预算、汇报进展、培训 |
| **核心结论** | 你最想传达的一个观点 | "我们应选择供应商 A" |

### 可选项

| 项目 | 说明 |
|:-----|:-----|
| 已有材料 | 数据、报告、之前的 PPT |
| 特殊要求 | 品牌色、页数限制、语言 |
| 行动号召 | 你希望听众听完后做什么？ |

> **Agent 注意**：若用户无法提供核心结论，先用 SCQR 框架帮 TA 梳理，再进入正式创作。

## 一、搭建演示文稿逻辑框架 Structuring Your Presentation

### 1.1 直击核心观点 Get to the Point

所有演示文稿的第一原则：**开篇即亮出核心结论，避免冗长铺垫。**

核心结论 = 直接回答观众心中最大的疑问。

### 1.2 帕累托 80/20 法则

#### 输入侧原则

- **20% 核心信息** → 支撑 **80% 核心成果**
- **80% 细节信息** → 仅起辅助作用

#### 演示准则

1. **服务核心目标**：所有内容必须服务核心目标，无关内容全部删减
2. **体现全面性**：展示全面性，但不要堆砌冗余细节
3. **时间压缩优先**：时间压缩场景优先保留 20% 核心内容
4. **把控节奏**：不仓促，保持与观众的连接

### 1.3 说服反直觉理论

#### 核心概念

| 术语 | 英文 | 说明 |
|:-----|:-----|:-----|
| **稀释效应** | Dilution Effect | 混杂弱势论据会削弱强论据的说服力 |
| **有效信息** | Diagnostic Information | 直接支撑论点的关键信息 |
| **无效冗余信息** | Non-diagnostic Information | 与论点无关或关联性弱的信息 |

#### 关键洞察

> **多条强论据组合，说服力远高于强弱论据混合。**

**实践指导**：

- ❌ 错误：3 条强论据 + 2 条弱论据（稀释效应降低说服力）
- ✅ 正确：3 条强论据（集中火力，说服力最大化）

### 1.4 金字塔逻辑框架 Pyramid Structure

#### 通用金字塔层级

```
┌─────────────────────────────────────┐
│     Governing Thought（顶层）        │
│     核心结论：直接回答观众心中核心疑问   │
├─────────────────────────────────────┤
│     Key Line（中层）                  │
│     核心分论点：联合佐证顶层结论         │
├─────────────────────────────────────┤
│     Support（底层）                   │
│     支撑素材：数据、事实、案例证据       │
└─────────────────────────────────────┘
```

**Agent 构建指令**：自上而下构建。检查每一层级是否回答"为什么 / 如何"类问题。若听众只听到顶层信息，就已足够理解你的立场。

#### 两类主流金字塔结构

##### （1）Applications-first / Inductive 先方案（归纳式）

| 层级 | 内容 | 示例 |
|:-----|:-----|:-----|
| **顶层** | Recommendation 最终建议 | "我们应选择供应商 A" |
| **中层** | Reason/Step 三大理由 / 落地步骤 | 成本最优、交付能力最强、技术适配性高 |
| **底层** | 配套数据、案例支撑 | TCO 对比表、历史交付数据、兼容性测试报告 |

##### （2）Principles-first / Deductive 先原则（演绎式）

| 层级 | 内容 | 示例 |
|:-----|:-----|:-----|
| **顶层** | Objective 核心目标 | "选择最优供应商" |
| **中层** | Issue/Criteria 三大评判标准 / 现存问题 | 成本、交付能力、技术适配性、售后支持 |
| **底层** | 对应佐证材料 | 各供应商在四个维度的详细对比数据 |

---

### 1.5 如何选择结构类型（适用场景）

#### 什么时候用"问题导向型"（Principles-first / Deductive）？

| 场景 | 说明 | 判断信号 |
|:-----|:-----|:---------|
| **方案不明确，需要集体决策** | 你自己也没完全想好，想带着团队一步步推导出最优解 | 开场白是"我们今天要探讨一个问题……" |
| **听众有抵触情绪，需要说服** | 一上来就给结论容易被质疑，带听众走一遍推理过程，让他们自己"推导"出结论，更容易接受 | 听众对议题有先入为主的反对意见 |
| **务实型听众** | 这类文化背景的人喜欢听故事、看过程，先给结论会觉得"太武断" | 听众期望理解完整逻辑链 |

#### 什么时候用"结论先行型"（Applications-first / Inductive）？

| 场景 | 说明 | 判断信号 |
|:-----|:-----|:---------|
| **听众是高管，时间极度稀缺** | CEO 或总监只有 5 分钟耐心，必须把结论和价值"砸"在他们脸上 | 会议时间 ≤15 分钟，听众日程紧凑 |
| **听众是逻辑严谨的分析型人群** | 德国/法国受众、技术专家、财务、工程师。他们觉得"先给结论，再用数据验证"才是高效专业的表现 | 听众背景偏技术、财务或工程 |
| **你拥有绝对权威，或结论显而易见** | 你是领域专家，或结论已板上钉钉（如合规整改），直接告知并解释理由即可 | 议题已有明确方向，无需讨论 |

#### 结构类型选择决策流程图

```
听众是否期望立即知道答案？
    │
    ├── 是 → 使用"结论先行型"（Applications-first）
    │
    └── 否 → 议题是否存在多重权衡且无绝对最优？
                │
                ├── 是 → 使用"问题导向型"（Principles-first）
                │
                └── 否 → 听众对背景了解较少？
                            │
                            ├── 是 → 使用"全局式框架"（Holistic）
                            │
                            └── 否 → 根据听众文化背景选择
                                    ├── 德法/欧洲 → 原则优先
                                    └── 英美/亚洲 → 方案优先
```

## 二、故事叙事技巧 Storytelling

### 2.1 为什么需要故事

1. **故事比数据更有说服力**：故事能够触发情感共鸣，增强记忆点
2. **放大产品/服务价值**：通过故事展示产品/服务的实际应用场景和效果
3. **个人真实案例更有效**：个人真实案例比群体泛化故事效果更好

### 2.2 3S 故事标准框架

| 阶段 | 英文 | 内容要求 | 写作指引 |
|:-----|:-----|:---------|:---------|
| **背景铺垫** | Setup | 事件发生时间、场景、你的身份与当时处境 | 建立代入感，让听众进入情境 |
| **矛盾冲突** | Struggle | 你/故事角色面临的核心难题、困境带来的影响 | 描绘困难和情绪反应（挫败、担忧、压力） |
| **解决与总结** | Solution | 问题如何解决、个人收获、该故事如何支撑本次演示核心目标 | 转折点、成功经验，提炼 Lesson |

#### 3S 故事示例

> **Setup**：在一次重大产品发布中，工程、运维、销售各自达成 KPI，但客户仍在等待。
>
> **Struggle**：一位客户直言："谁真正负责我的端到端体验？"——这就是冲突。我们意识到孤岛在内部不失败，但在外部客户体验处失败。
>
> **Solution**：我们引入共享客户旅程指标，用联合问责替代孤岛 KPI。一个季度内交接减少，上线时间显著改善。**Lesson**：孤岛 KPI 导致局部最优而非全局最优。

### 2.3 故事评判维度

| 维度 | 说明 | 评判标准 |
|:-----|:-----|:---------|
| **画面感 / 场景完整度** | 听众能否在脑海中形成清晰画面 | 场景具体、细节丰富、有时间地点 |
| **逻辑通顺度** | 故事发展是否符合逻辑 | 因果关系清晰、转折自然、无跳跃 |
| **情绪感染力** | 能否引发听众情感共鸣 | 冲突真实、情绪具体、有代入感 |

> **Agent 使用指南**：在 PPT 的"问题"或"冲突"页嵌入 3S 故事，能显著提升说服力。

## 三、可视化素材设计 Working with Visuals

### 3.1 四大可视化设计核心原则

| 原则 | 英文 | 说明 | Agent 操作指引 |
|:-----|:-----|:-----|:---------------|
| **放大重点** | Make it Big | 突出关键信息 | 标题字号 ≥28pt，关键数字 ≥36pt，确保 2 米外可读 |
| **保持简洁** | Keep it Simple | 剔除冗余文字 | 每页最多 1 个核心观点，正文 ≤6 行，图表去掉网格线和不必要图例 |
| **逻辑清晰** | Make it Clear | 层级分明 | 标题必须是结论式短语，关键数据用颜色或加粗突出 |
| **分层递进** | Be Progressive | 分批次释放内容 | 避免一屏堆砌全部信息，复杂流程拆分为多页 |

### 3.2 递进式幻灯片设计逻辑

#### 四层递进结构

| 层级 | 内容 | 说明 |
|:-----|:-----|:-----|
| **第一层** | 仅展示大类标题 | 让听众了解整体框架和主题 |
| **第二层** | 补充大类基础描述 | 提供基本概念和背景信息 |
| **第三层** | 补充约束条件、落地条款 | 细化执行细节和限制条件 |
| **第四层** | 补充潜在风险、全局连锁影响 | 深入分析风险和影响范围 |

#### 递进原则执行方式

| 场景 | 执行方式 |
|:-----|:---------|
| **静态 PPT**（PDF、图片导出） | 将复杂流程拆成多页，每页聚焦一个阶段，用页码序号暗示进度（如"3/7"） |
| **支持动画的 PPT**（PowerPoint、Keynote） | 使用逐步出现的动画，每点击一次揭示一个要素 |
| **Agent 生成的 PPT** | 默认采用"多页拆分"方式；若用户指定工具支持动画，再使用动画方案 |

#### 递进原则示例（数据迁移流程）

- **页 1/4**：现状分析 — 源系统数据量、质量评估
- **页 2/4**：迁移方案 — ETL 管道设计、增量同步策略
- **页 3/4**：执行计划 — 时间线、回滚方案、验证节点
- **页 4/4**：风险控制 — 应急预案、数据一致性校验

## 四、页面设计规范 Slide Design

### 4.1 动作标题 Action Title（Dot）

**定义**：一个完整的、结论式的、与决策相关的短句。它回答"所以呢？（So what?）"这个问题，而非描述页面内容。

**规则**：每页 PPT 的标题必须是 **Dot**（动作标题），正文为 **Dash**（支撑证据）。整个 PPT 的所有 Dot 串联起来应能独立讲述完整故事。

| ❌ 错误（描述性标题） | ✅ 正确（结论式标题） |
|:---------------------|:---------------------|
| 市场数据 | 市场份额下降 5%，需调整定价策略 |
| 技术架构图 | 微服务架构将部署时间从 2 周缩短至 2 天 |
| 团队规模 | 团队扩张 40% 但产出仅增长 15%，效率待优化 |
| 竞品分析 | 竞品 A 已抢占 30% 高端市场，我司需加速产品迭代 |
| Q3 销售数据 | Q3 销售额增长 12%，但利润率下降 3 个百分点 |
| 客户反馈总结 | 客户满意度下降 8 分，需在 Q4 启动体验优化专项 |

### 4.2 水平流与垂直流检查

- **水平流**：按顺序阅读每页的"动作标题"，看是否能构成连贯故事。若不能，需调整标题措辞。
- **垂直流**：每页的正文是否有力支撑该页标题？是否有多余信息或缺失关键证据？

> **Agent 自检**：只看所有页面标题，能否明白全部结论？若不能，则标题不够清晰。

### 4.3 页面标准结构

| 页面类型 | 必需性 | 内容要求 |
|:---------|:-------|:---------|
| **标题页** | 必需 | 演示标题、副标题、作者/团队、日期 |
| **目录页** | 可选 | 页数 >10 时建议包含；列出主要章节 |
| **核心信息页** | 必需 | 金字塔顶端的一句话结论（首页或第 2 页） |
| **正文页** | 必需 | 每页一个 Dot + Dash，页码标注（当前页/总页数） |
| **总结/行动号召页** | 必需 | 结尾五要素（见第五章） |
| **附录** | 可选 | 详细数据、方法论说明、参考文献 |

### 4.4 页数建议

| 场景 | 建议页数 | 说明 |
|:-----|:---------|:-----|
| 高管决策会（10-15 分钟） | 5-8 页 | 不含讨论时间，每页约 2 分钟 |
| 跨部门提案（30 分钟） | 10-15 页 | 含互动时间 |
| 项目进展汇报 | 8-12 页 | 含数据图表 |
| 详细研究报告 | 20-30 页 | 附录另算 |

## 五、结尾规范 Closing

### 5.1 结尾五要素

| 要素 | 作用 | 写作指引 |
|:-----|:-----|:---------|
| **信号** | 提示听众即将收尾 | "总之""在结束前""综合以上分析" |
| **总结** | 复述 3 个核心观点 | 对应一级支撑论点，每点一句话 |
| **重申目的** | 回归演示的根本目的 | "这项投资的根本目的是确保…" |
| **行动号召** | 明确下一步 | 具体动作 + 负责人 + 截止日期 |
| **结束语** | 礼貌收尾 | 感谢、邀请提问、说明后续沟通方式 |

### 5.2 结尾模板

```
【信号】总之，基于以上分析，我们提出了明确的方向。

【总结】
第一，[核心观点 1] — [一句话支撑]。
第二，[核心观点 2] — [一句话支撑]。
第三，[核心观点 3] — [一句话支撑]。

【重申目的】这项工作的根本目的是 [WHY]，这将直接影响 [关键业务指标]。

【行动号召】请各位在 [具体日期] 前完成 [具体动作]。
我们将在 [日期] 安排 [后续会议/交付物]。

【结束语】感谢各位的时间。如有疑问，欢迎现在提出，
也可会后通过 [联系方式] 与我们沟通。
```

### 5.3 结尾反模式

| ❌ 错误写法 | 问题 | ✅ 正确写法 |
|:-----------|:-----|:-----------|
| "谢谢！" | 缺少行动号召，听众不知道下一步 | "请在周五前签署预算审批表" |
| "以上就是我的汇报" | 没有总结，信息未强化 | 复述 3 个核心观点 + 行动号召 |
| "希望对大家有帮助" | 模糊、被动、无责任归属 | "请市场部在周三前提供竞品数据" |

## 六、Agent 创作速查清单 Checklist

在最终输出 PPT 前，Agent 应逐项确认：

### 输入阶段

- [ ] 已确认听众类型、汇报时长、核心诉求
- [ ] 已确认用户是否提供了核心结论（若无，先用 SCQR 梳理）

### 结构阶段

- [ ] 根据情景选定了合适的结构类型（问题导向 / 结论先行 / 全局式）
- [ ] 首页包含清晰的核心目的（WHY）
- [ ] 金字塔顶端信息明确，一级支撑 3~5 个，二级有数据

### 内容阶段

- [ ] 每页标题是结论式动作标题（Dot），正文是支撑证据（Dash）
- [ ] 水平流：所有标题连读是一个完整故事
- [ ] 垂直流：每页内容完全支撑本页标题
- [ ] 无弱势论据混入（避免稀释效应）
- [ ] 结尾包含明确的行动号召（动作 + 负责人 + 截止日期）

### 视觉阶段

- [ ] 标题字号 ≥28pt，关键数字 ≥36pt
- [ ] 每页 ≤6 行正文，图表去掉网格线
- [ ] 复杂流程已按渐进原则拆分为多页
- [ ] 避免 3D 图表、过多颜色（最多 3 种主色 + 灰色系）

### 输出阶段

- [ ] 包含标题页、正文页、总结/行动号召页
- [ ] 页码标注完整（当前页/总页数）
- [ ] 总页数符合场景建议

## 七、常见错误与纠正 Anti-patterns

### 7.1 标题类错误

| 错误类型 | 错误示例 | 正确示例 | 问题根因 |
|:---------|:---------|:---------|:---------|
| 描述性标题 | "Q3 销售数据" | "Q3 销售额增长 12%，但利润率下降 3 个百分点" | 未回答"So what?" |
| 过于宽泛 | "技术方案" | "采用微服务架构，预计降低 40% 部署时间" | 缺乏具体结论 |
| 无行动导向 | "客户反馈总结" | "客户满意度下降 8 分，需在 Q4 启动体验优化专项" | 缺少决策关联 |

### 7.2 结构类错误

| 错误类型 | 问题描述 | 纠正方式 |
|:---------|:---------|:---------|
| 一页多观点 | 单页同时讨论成本、风险、时间线 | 拆分为 3 页，每页聚焦一个维度 |
| 证据与标题脱节 | 标题说"成本最优"，正文展示的是技术架构 | 确保垂直流：正文必须直接支撑标题 |
| 结论出现过晚 | 前 8 页都是背景铺垫，第 9 页才给建议 | 使用结论先行结构，首页即给出核心建议 |
| 缺少逻辑连接 | 页与页之间跳跃，读者跟不上 | 添加过渡句或预览页（"接下来从三个维度展开"） |

### 7.3 视觉类错误

| 错误类型 | 问题描述 | 纠正方式 |
|:---------|:---------|:---------|
| 文字过多 | 单页 >10 行文字 | 精简至 ≤6 行，多余内容移至附录 |
| 图表过小 | 图表占据页面不到 1/3 | 图表应占据页面 60% 以上 |
| 颜色过多 | 使用 >5 种颜色 | 统一配色方案，最多 3 种主色 + 灰色系 |
| 字号不一致 | 标题 24pt、正文 18pt、注释 10pt | 建立字号层级：标题 28-36pt、正文 18-24pt、注释 12-14pt |

### 7.4 说服力错误

| 错误类型 | 问题描述 | 纠正方式 |
|:---------|:---------|:---------|
| 强弱论据混搭 | 3 条强论据 + 2 条弱论据 | 只保留强论据，删除弱论据（稀释效应） |
| 缺少数据支撑 | 只有观点没有证据 | 每个论点至少配 1 个数据/案例 |
| 故事缺失 | 纯数据堆砌，缺乏情感连接 | 在关键节点嵌入 3S 故事 |

## 八、迭代修改指南

当用户反馈需要修改时，按以下优先级处理：

| 修改类型 | 操作方式 | 影响范围 |
|:---------|:---------|:---------|
| **改标题** | 直接修改动作标题，确保仍为结论式 | 仅当前页 + 水平流检查 |
| **调整结构** | 重新排列页面顺序，检查逻辑连接 | 全局水平流，需重新验证 |
| **补充证据** | 在指定页添加数据/案例 | 仅当前页垂直流 |
| **增删页面** | 添加或移除页面，调整页码 | 全局结构、页数、结尾位置 |
| **风格调整** | 修改配色、字号、图表样式 | 视觉层，不影响逻辑 |

> **Agent 注意**：每次修改后，重新运行速查清单中的"内容阶段"和"视觉阶段"检查。

## 附录：中英对照术语表

| 中文 | 英文 | 定义 |
|:-----|:-----|:-----|
| 核心结论 | Governing Thought | 金字塔顶层，直接回答观众心中核心疑问 |
| 核心分论点 | Key Line | 金字塔中层，联合佐证顶层结论 |
| 支撑素材 | Support | 金字塔底层，数据、事实、案例证据 |
| 稀释效应 | Dilution Effect | 混杂弱势论据会削弱强论据的说服力 |
| 有效信息 | Diagnostic Information | 直接支撑论点的关键信息 |
| 无效冗余信息 | Non-diagnostic Information | 与论点无关或关联性弱的信息 |
| 背景铺垫 | Setup | 3S 故事框架第一阶段 |
| 矛盾冲突 | Struggle | 3S 故事框架第二阶段 |
| 解决与总结 | Solution | 3S 故事框架第三阶段 |
| 动作标题 | Action Title (Dot) | 完整的、结论式的、与决策相关的短句 |
| 支撑证据 | Dash | 支撑 Dot 的数据、事实、案例 |
| 水平流 | Horizontal Flow | 所有标题连读构成完整故事 |
| 垂直流 | Vertical Flow | 每页内容支撑本页标题 |
