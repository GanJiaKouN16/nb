# AI Coding Agent 成本优化

## 核心结论
**AI Coding Agent 的成本大头不是用户输入的提问，而是系统为回答问题而重复搬运的大量上下文。** 成本优化的本质是减少系统重复的无效工作。

---

## 一、成本构成
用户提问在每次请求中占比不足 1%，真正的成本来自系统自动附加的固定前缀、会话历史、运行时检索和工具往返。

| 成本类型 | 核心问题 |
| :--- | :--- |
| **输入 Token** | 每轮重复携带，占比最大 |
| **输出 Token** | 回答冗长导致失控 |
| **推理 Token** | 简单任务用高推理档位导致溢价 |
| **工具往返成本** | 工具调用链产生多段上下文交换 |
| **重试成本** | 第一次不合格导致整包上下文反复付费 |

---

## 二、五层优化路径（从易到难、从便宜到贵）

### 第一层：使用习惯（零工程投入）
- 一个 Session 只服务一个目标，避免会话无限拉长
- 长会话及时压缩（`/compact`），历史不是资产而是负债
- 长期信息外置到文档/Memory，会话只承载当前状态
- 指定输出格式（直接给结论/JSON），同时减少输出和重试
- Skill 按需加载，低频不常驻
- MCP 按需常驻，工具越多决策越慢、定义越重
- 能 CLI 解决的优先 CLI，比 MCP 更轻量
- 引用文件带完整路径（`@路径`），省去搜索链路
- 一次把意图说完，避免多轮来回确认

### 第二层：模型路由（让合适模型干合适活）
- 按任务分档：复杂→强模型，简单→便宜模型，重复→稳定模型
- 先过便宜模型做初筛，复杂时再升级
- 调整推理预算旋钮（`reasoning effort` / `thinking budget`），简单任务不用深度思考
- Skill / Agent / Command 绑定默认模型，成本治理系统化

### 第三层：上下文压缩（压减必然进入上下文的内容）
四种工具可叠加使用：

| 工具 | 压缩对象 | 典型节省 |
| :--- | :--- | :--- |
| **RTK** | 终端命令输出 | 89% |
| **Caveman** | AI 回复输出 | 65-75% |
| **headroom** | 所有进上下文的内容 | 47-92% |
| **context-mode** | MCP 工具结果 | 98%（工具输出） |

### 第四层：代码图谱（让 AI 不再盲搜）
- 真正贵的是“找代码”而非“读代码”，图谱让 AI 在读文件前就知道该读哪里
- **Graphify**：用 Tree-sitter 构建知识图谱，比直接读文件减少 **71.5 倍** Token
- **CodeGraph**：MCP Server + 图数据库，大型仓库平均节省 **16-40%** 成本

### 第五层：多 Agent 协作（任务边界清晰）
- 单 Agent 越用越贵：规划、写代码、测试、Review 全在一个会话，上下文越来越胖
- **subagent 任务隔离**：子任务独立上下文，主 session 只看结果
- **Orchestrator-Worker 模式**：协调器只负责规划调度，Worker 只看当前步骤所需上下文，每轮成本压缩 **5-10 倍**
- 上下文隔离后通过共享外置文件（JSON）传递数据，不通过会话历史
- 独立子任务并行执行，时间和成本同时压缩

---

## 三、六大常见误区
1. **上下文越多越好** → 噪音越多，模型越容易分神
2. **MCP 越多越强** → 低频工具长期挂载，贡献更多成本
3. **所有 Agent 都上最强模型** → 放弃路由，直接烧钱
4. **聊天记录当长期记忆** → 成本越来越高，关键信息难提取
5. **只看单价不看总成本** → 便宜模型导致重试多，总成本未必低
6. **Prompt 越短越好** → 压缩丢必要信息导致重试，压缩要精准
