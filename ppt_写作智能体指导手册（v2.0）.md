# PPT 写作智能体指导手册（v2.0）
## 面向商务/技术汇报的结构化演示设计指南

本手册整合了麦肯锡金字塔原理、SCQR故事框架、三种逻辑结构及视觉设计四原则，旨在指导 AI Agent 或人类写作者快速生成逻辑清晰、说服力强的PPT。请按以下步骤和情景选择进行创作。

---

## 〇、创作前：信息收集（Agent 输入规范）

在开始创作前，Agent 应向用户确认以下信息：

| 必填项 | 说明 | 示例 |
|--------|------|------|
| **听众** | 谁会看/听这个演示？ | 高管团队、技术同事、客户 |
| **时长** | 汇报时间（不含讨论） | 10 分钟、30 分钟 |
| **核心诉求** | 演示的根本目的是什么？ | 说服批准预算、汇报进展、培训 |
| **核心结论** | 你最想传达的一个观点 | "我们应选择供应商 A" |

| 可选项 | 说明 |
|--------|------|
| 已有材料 | 数据、报告、之前的 PPT |
| 特殊要求 | 品牌色、页数限制、语言 |
| 行动号召 | 你希望听众听完后做什么？ |

> **智能体注意**：若用户无法提供核心结论，先用 SCQR 框架帮 TA 梳理，再进入正式创作。

## 一、开始前：情景选择（决定故事框架）

在动笔前，先根据 **听众类型**、**决策紧迫度** 和 **议题复杂度** 选择合适的故事结构：

| 情景编号 | 典型场景 | 推荐结构 | 理由 |
|---------|---------|---------|------|
| **A** | 向高管汇报，时间紧迫（如10分钟内的决策会） | **结构 B：应用优先** | 先给建议，再解释原因，直接回答"我应该做什么" |
| **B** | 跨部门协作提案，议题复杂，涉及多个利益相关方 | **结构 C：原则优先** | 先确立共同决策标准，再按标准评估选项，减少分歧 |
| **C** | 项目进展汇报，听众熟悉背景，需要完整理解 | **结构 A：整体式** | 先展示全面证据，再自然引出结论，适合教育性汇报 |
| **D** | 需要情感共鸣或改变听众固有观念（如变革管理） | **SCQR + 3S故事法** | 用故事制造张力（冲突），再引出解决方案，打动人心 |
| **E** | 数据密集型汇报（如年度业绩） | **金字塔原理 + 水平/垂直流** | 顶部结论先行，下层分组数据，每页标题传递核心信息 |

**决策流程图**：
- 听众是否期望立即知道答案？ → **是** → 使用 **结构B**
- 议题是否存在多重权衡且无绝对最优？ → **是** → 使用 **结构C**
- 是否需要铺垫背景以说服听众？ → **是** → 使用 **结构A**
- 是否需要激发情感或改变认知？ → **是** → 使用 **SCQR + 3S**

## 二、核心方法详解（按创作顺序）

### 步骤1：撰写故事线（Storyline）

#### 1.1 使用 SCQR 构建开场逻辑
每条故事线应从 **情境（S）** 开始，引入 **冲突（C）**，提出 **问题（Q）**，最终给出 **解决方案（R）**。这是吸引听众注意力的引擎。

**写作模板**：
- **S（情境）**：当前已知的稳定背景（时间、地点、业务状态）。
- **C（冲突）**：发生了何种变化、风险、瓶颈或意料之外的挑战。
- **Q（问题）**：由此引发的核心决策问题（用"如何""是否"等句式）。
- **R（解决方案）**：您的明确建议或行动计划。

**强弱对比示例（以IT运维为例）**：

| 弱版（低效） | 强版（高管就绪） |
|-------------|----------------|
| S：IT系统变得复杂。<br>C：事故增多。<br>Q：如何改进？<br>R：优化流程。 | S：公司全球数字平台扩展，IT运维支撑多地业务。<br>C：事故量同比增长20%，平均修复时间（MTTR）上升，常规维护占用70%团队精力。<br>Q：我们应如何在不影响业务扩张的前提下，系统性地降低运维风险？<br>R：建议实施自动化运维平台，预计将MTTR降低40%，并释放30%人力用于创新。 |

> **智能体注意**：强版必须包含**量化数据**、**具体威胁**和**可衡量的收益**。

#### 1.2 何时使用 3S 模型扩展冲突

3S（Setup-Struggle-Solution）是 SCQR 中"冲突（C）"的深度展开版本，用于增强情感张力。

**判断条件**：

| 使用纯 SCQR | 使用 SCQR + 3S |
|-------------|----------------|
| 数据驱动、逻辑说服 | 需要情感共鸣、改变固有观念 |
| 常规汇报、进展更新 | 变革管理、危机沟通 |
| 听众理性、专业 | 听众有抵触情绪或路径依赖 |

**3S 写作框架**：
- **Setup**：描述时间、地点、人物，建立代入感。
- **Struggle**：详细描绘困难和情绪反应（挫败、担忧、压力），以及错误尝试。
- **Solution**：转折点、成功经验，并提炼出"点"（Lesson）。

**示例**：
> "在一次重大产品发布中，工程、运维、销售各自达成KPI，但客户仍在等待。一位客户直言：'谁真正负责我的端到端体验？'——这就是冲突。我们后来引入共享客户旅程指标，用联合问责替代孤岛KPI，一个季度内交接减少，上线时间显著改善。点：孤岛在内部不失败，但在外部客户体验处失败。"

将此故事放在PPT的"问题"或"冲突"页，能显著提升说服力。

#### 1.3 使用 Dot-Dash 结构展开后续页面
在确定主故事线后，用 **点（Dot）** 定义每一页的核心信息（即动作标题），用 **划（Dash）** 列出支撑该点的证据。

- **Dot**：一句完整的决策相关观点（如"亚太区需增加30%云容量"）。
- **Dash**：数据图表、案例、逻辑推理、引用等（如"过去6个月使用量增长趋势""供应商报价对比"）。

**规则**：每页PPT的标题必须是 **Dot**，正文为 **Dash**；整个PPT的所有 **Dot** 串联起来应能独立讲述完整故事（即水平流）。

**动作标题（Action Title）定义**：一个完整的、结论式的、与决策相关的短句。它回答"所以呢？（So what?）"这个问题，而非描述页面内容。

| 错误（描述性） | 正确（结论式） |
|---------------|---------------|
| 市场数据 | 市场份额下降 5%，需调整定价策略 |
| 技术架构图 | 微服务架构将部署时间从 2 周缩短至 2 天 |
| 团队规模 | 团队扩张 40% 但产出仅增长 15%，效率待优化 |
| 竞品分析 | 竞品 A 已抢占 30% 高端市场，我司需加速产品迭代 |

### 步骤2：应用金字塔原理（Pyramid Principle）组织层次

金字塔原理要求 **自上而下** 书写：

1. **顶端（核心信息）**：一句话概括您的最终建议或核心洞察（不超过20字）。
2. **一级支撑（关键论点）**：3~5个理由/方面，每个支撑顶端论点。
3. **二级支撑（证据层）**：为每个一级论点提供数据、案例或逻辑细节。

**示例**：
- **顶端**：我们应选择供应商A。
- **一级论点1**：成本最优 → 二级：总拥有成本（TCO）低于B 15%。
- **一级论点2**：交付能力最强 → 二级：过往项目准时交付率100%。
- **一级论点3**：技术适配性高 → 二级：完全兼容现有架构。

> **智能体指令**：检查每一层级是否回答"为什么/如何"类问题。若听众只听到顶端信息，就已足够理解您的立场。

### 步骤3：选择三种逻辑结构之一（详见情景选择）

三种结构的具体写作框架如下：

#### 结构 A：整体式（Holistic）—— 用于背景完整型汇报
- **顺序**：
  1. 主题A证据 → 2. 主题B证据 → 3. 主题C证据 → 4. 汇总结论 → 5. 最终建议
- **适用**：教育训练、详细复盘、听众已预留充足时间。
- **优点**：逻辑递进自然，容易理解。
- **缺点**：结论出现较晚。

#### 结构 B：应用优先（Application-first）—— 用于决策驱动型汇报
- **顺序**：
  1. **明确建议（推荐）** → 2. 预览关键结论（A, B, C） → 3. 分别展示A、B、C的证据 → 4. 重申建议
- **适用**：高管会议、预算审批、紧急决策。
- **优点**：立即聚焦，节省时间。
- **缺点**：可能缺乏背景铺垫，需在前几句快速弥补。

#### 结构 C：原则优先（Principles-first）—— 用于复杂权衡型汇报
- **顺序**：
  1. 明确决策原则/标准 → 2. 按原则评估各选项 → 3. 推荐最优选项并解释符合原则之处
- **适用**：战略选择、投资组合管理、多方案比较。
- **优点**：建立客观共识，减少争议。
- **缺点**：需要提前定义原则，工作量较大。

**结构 C 的原则选取指南**：

原则应从以下维度中选取 3~5 个（过多会分散注意力）：

| 维度 | 适用场景 | 示例原则 |
|------|---------|---------|
| 成本 | 预算敏感、采购决策 | 总拥有成本（TCO）最低 |
| 风险 | 合规、安全、供应链 | 风险敞口最小化 |
| 可扩展性 | 技术架构、业务增长 | 支持未来 3 年 10x 增长 |
| 时间 | 紧急项目、市场窗口 | 6 个月内可落地 |
| 用户体验 | 产品设计、服务改进 | 用户满意度提升 ≥20% |
| 合规性 | 金融、医疗、跨境业务 | 满足 GDPR/等保要求 |

**原则选取的注意事项**：
- 原则之间应尽量正交（不重叠）。
- 原则应被所有利益相关方认可（这是减少争议的前提）。
- 每个原则应可量化或可明确判断（避免"感觉好"这类模糊标准）。

**结构 C 完整示例（供应商选择）**：

```
【页1】标题：基于四项标准评估，推荐供应商 A
正文：评估维度 — 成本、交付能力、技术适配性、售后支持

【页2】标题：成本维度 — A 的 TCO 低于 B 达 15%
正文：三年总拥有成本对比图表...

【页3】标题：交付维度 — A 过往准时交付率 100%
正文：历史项目交付数据...

【页4】标题：技术维度 — A 完全兼容现有架构，零改造成本
正文：技术兼容性矩阵...

【页5】标题：售后维度 — A 提供 7×24 本地支持，响应时间 ≤2h
正文：SLA 对比表...

【页6】标题：综合四项标准，A 是最优选择
正文：雷达图汇总 + 建议下一步
```

**智能体判断逻辑**：
- 若听众**明确要求结论** → 强制使用 **结构B**。
- 若议题存在**多个相互冲突的目标** → 优先考虑 **结构C**。
- 若听众对背景**了解较少** → 可采用 **结构A**，但应压缩背景部分。

### 步骤4：设计每页幻灯片的视觉与逻辑

#### 4.1 水平流与垂直流检查
- **水平流**：按顺序阅读每页的"动作标题（Dot）"，看是否能构成连贯故事。若不能，需调整标题措辞。
- **垂直流**：每页的正文（Dash）是否有力支撑该页标题？是否有多余信息或缺失关键证据？

#### 4.2 应用视觉设计四原则（Make it Big, Simple, Clear, Progressive）

| 原则 | 具体操作 |
|------|---------|
| **放大（Big）** | 标题字号 ≥28pt，关键数字 ≥36pt，确保从 2 米外可读。图表标题 ≥18pt，数据标签 ≥12pt。 |
| **简洁（Simple）** | 每页最多 1 个核心观点，正文 ≤6 行，图表去掉网格线和不必要图例。避免 3D 图表（增加视觉噪音）。 |
| **清晰（Clear）** | 标题必须是一个**完整的结论式短语**（如"云成本同比上升15%，需优化架构"），而非中性描述（如"云成本数据"）。关键数据点用颜色或加粗突出。 |
| **渐进（Progressive）** | 复杂过程分多页展示，每页引入一个新元素。具体操作见下方说明。 |

**渐进原则的执行方式**：

| 场景 | 执行方式 |
|------|---------|
| 静态 PPT（PDF、图片导出） | 将复杂流程拆成多页，每页聚焦一个阶段，用页码序号暗示进度（如"3/7"），或用颜色渐变标识阶段 |
| 支持动画的 PPT（PowerPoint、Keynote） | 使用逐步出现的动画，每点击一次揭示一个要素 |
| Agent 生成的 PPT | 默认采用"多页拆分"方式；若用户指定工具支持动画，再使用动画方案 |

**示例（渐进原则应用）**：

将"数据迁移流程"拆为 4 页：
- 页 1/4：现状分析 — 源系统数据量、质量评估
- 页 2/4：迁移方案 — ETL 管道设计、增量同步策略
- 页 3/4：执行计划 — 时间线、回滚方案、验证节点
- 页 4/4：风险控制 — 应急预案、数据一致性校验

> **智能体自检**：浏览PPT时，只看标题是否能明白全部结论？若不能，则标题不够清晰。

### 步骤5：撰写结尾（Closing with Call to Action）

结尾应包含 5 个要素（信号、总结、重申目的、行动号召、结束语）：

| 要素 | 作用 | 写作指引 |
|------|------|---------|
| **信号** | 提示听众即将收尾 | "总之""在结束前""综合以上分析" |
| **总结** | 复述 3 个核心观点 | 对应一级支撑论点，每点一句话 |
| **重申目的** | 回归演示的根本目的 | "这项投资的根本目的是确保…" |
| **行动号召** | 明确下一步 | 具体动作 + 负责人 + 截止日期 |
| **结束语** | 礼貌收尾 | 感谢、邀请提问、说明后续沟通方式 |

**结尾模板**：

```
【信号】总之，基于以上分析，我们提出了明确的方向。

【总结】
第一，[核心观点 1] — [一句话支撑]。
第二，[核心观点 2] — [一句话支撑]。
第三，[核心观点 3] — [一句话支撑]。

【重申目的】这项工作的根本目的是 [WHY]，这将直接影响 [关键业务指标]。

【行动号召】请各位在 [具体日期] 前完成 [具体动作]。
我们将在 [日期] 安排 [后续会议/交付物]。

【结束语】感谢各位的时间。如有疑问，欢迎现在提出，
也可会后通过 [联系方式] 与我们沟通。
```

**结尾反模式**：

| 错误写法 | 问题 | 正确写法 |
|---------|------|---------|
| "谢谢！" | 缺少行动号召，听众不知道下一步 | "请在周五前签署预算审批表" |
| "以上就是我的汇报" | 没有总结，信息未强化 | 复述 3 个核心观点 + 行动号召 |
| "希望对大家有帮助" | 模糊、被动、无责任归属 | "请市场部在周三前提供竞品数据" |

## 四、输出格式规范（Agent 输出要求）

Agent 生成的 PPT 应包含以下标准页面结构：

| 页面类型 | 必需性 | 内容要求 |
|---------|--------|---------|
| **标题页** | 必需 | 演示标题、副标题、作者/团队、日期 |
| **目录页** | 可选 | 页数 >10 时建议包含；列出主要章节 |
| **核心信息页** | 必需 | 金字塔顶端的一句话结论（首页或第 2 页） |
| **正文页** | 必需 | 每页一个 Dot + Dash，页码标注 |
| **总结/行动号召页** | 必需 | 结尾五要素 |
| **附录** | 可选 | 详细数据、方法论说明、参考文献 |

**页码规范**：所有正文页右下角标注页码，格式为"当前页/总页数"。

## 五、常见错误与纠正示例（反模式清单）

Agent 在创作过程中应主动避免以下错误：

### 5.1 标题类错误

| 错误类型 | 错误示例 | 正确示例 | 问题根因 |
|---------|---------|---------|---------|
| 描述性标题 | "Q3 销售数据" | "Q3 销售额增长 12%，但利润率下降 3 个百分点" | 未回答"So what?" |
| 过于宽泛 | "技术方案" | "采用微服务架构，预计降低 40% 部署时间" | 缺乏具体结论 |
| 无行动导向 | "客户反馈总结" | "客户满意度下降 8 分，需在 Q4 启动体验优化专项" | 缺少决策关联 |

### 5.2 结构类错误

| 错误类型 | 问题描述 | 纠正方式 |
|---------|---------|---------|
| 一页多观点 | 单页同时讨论成本、风险、时间线 | 拆分为 3 页，每页聚焦一个维度 |
| 证据与标题脱节 | 标题说"成本最优"，正文展示的是技术架构 | 确保垂直流：正文必须直接支撑标题 |
| 结论出现过晚 | 前 8 页都是背景铺垫，第 9 页才给建议 | 使用结构 B，首页即给出核心建议 |
| 缺少逻辑连接 | 页与页之间跳跃，读者跟不上 | 添加过渡句或预览页（"接下来从三个维度展开"） |

### 5.3 视觉类错误

| 错误类型 | 问题描述 | 纠正方式 |
|---------|---------|---------|
| 文字过多 | 单页 >10 行文字 | 精简至 ≤6 行，多余内容移至附录 |
| 图表过小 | 图表占据页面不到 1/3 | 图表应占据页面 60% 以上 |
| 颜色过多 | 使用 >5 种颜色 | 统一配色方案，最多 3 种主色 + 灰色系 |
| 字号不一致 | 标题 24pt、正文 18pt、注释 10pt | 建立字号层级：标题 28-36pt、正文 18-24pt、注释 12-14pt |

## 六、智能体创作速查清单（Checklist）

在最终输出PPT前，请确认以下项目：

**输入阶段**：
- [ ] 已确认听众类型、汇报时长、核心诉求。
- [ ] 已确认用户是否提供了核心结论（若无，先用 SCQR 梳理）。

**结构阶段**：
- [ ] 根据情景选定了合适的结构（A/B/C/D/E）。
- [ ] 首页包含清晰的 WHY（演示核心目的）。
- [ ] 故事线使用 SCQR 开头，且冲突具体、量化。
- [ ] 若使用结构 C，已选取 3-5 个正交、可量化的决策原则。

**内容阶段**：
- [ ] 每页标题是结论式 Dot，正文是 Dash 证据。
- [ ] 金字塔顶端信息明确，一级支撑 3~5 个，二级有数据。
- [ ] 水平流：所有标题连读是一个完整故事。
- [ ] 垂直流：每页内容完全支撑本页标题。
- [ ] 结尾包含明确的行动号召（动作 + 负责人 + 截止日期）。

**视觉阶段**：
- [ ] 标题字号 ≥28pt，关键数字 ≥36pt。
- [ ] 每页 ≤6 行正文，图表去掉网格线。
- [ ] 复杂流程已按渐进原则拆分为多页。

**输出阶段**：
- [ ] 包含标题页、正文页、总结/行动号召页。
- [ ] 页码标注完整（当前页/总页数）。
- [ ] 总页数符合场景建议（见下方）。

**页数建议**：

| 场景 | 建议页数 | 说明 |
|------|---------|------|
| 高管决策会（10-15 分钟汇报） | 5-8 页 | 不含讨论时间，每页约 2 分钟 |
| 跨部门提案（30 分钟汇报） | 10-15 页 | 含互动时间 |
| 项目进展汇报 | 8-12 页 | 含数据图表 |
| 详细研究报告 | 20-30 页 | 附录另算 |

## 七、迭代修改指南

当用户反馈需要修改时，按以下优先级处理：

| 修改类型 | 操作方式 | 影响范围 |
|---------|---------|---------|
| **改标题** | 直接修改动作标题，确保仍为结论式 | 仅当前页 + 水平流检查 |
| **调整结构** | 重新排列页面顺序，检查逻辑连接 | 全局水平流，需重新验证 |
| **补充证据** | 在指定页添加数据/案例 | 仅当前页垂直流 |
| **增删页面** | 添加或移除页面，调整页码 | 全局结构、页数、结尾位置 |
| **风格调整** | 修改配色、字号、图表样式 | 视觉层，不影响逻辑 |

> **智能体注意**：每次修改后，重新运行速查清单中的"内容阶段"和"视觉阶段"检查。
